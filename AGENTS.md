# 搭把手项目协作约束文件

## 重要说明

本文件是搭把手项目的核心约束文件，所有参与开发的Agent和开发者必须严格遵守。
违反约束的代码提交将被拒绝合并。

## 一、项目架构约束

### 1.1 模块边界约束
- **禁止跨模块直接调用**：模块间必须通过定义好的API接口通信
- **禁止修改他人模块**：每个模块有明确的负责人，未经授权不得修改
- **新增模块必须审批**：新增模块需要提交RFC文档并获得核心开发者批准

### 1.2 目录结构约束
```
backend/
├── dabashou-common/        # 公共模块（仅包含工具类、常量、异常定义）
├── dabashou-system/        # 系统管理模块（仅系统管理员可访问）
├── dabashou-user/          # 用户模块（用户注册、登录、个人信息）
├── dabashou-skill/         # 技能模块（技能分类、标签、用户技能）
├── dabashou-shelf/         # 技能货架模块（服务发布、管理）
├── dabashou-demand/        # 需求模块（需求发布、匹配）
├── dabashou-order/         # 订单模块（订单状态机、核销码）
├── dabashou-point/         # 积分模块（积分管理、担保池）
├── dabashou-message/       # 消息模块（即时通讯、通知）
├── dabashou-credit/        # 信用评价模块（评价、违规、申诉）
├── dabashou-stat/          # 数据统计模块（统计分析）
├── dabashou-admin/         # 管理后台模块
└── dabashou-api/           # API网关模块（聚合所有接口）
```

### 1.3 命名规范约束
- **包名**：`com.dabashou.{module}`，例如：`com.dabashou.user`
- **类名**：大驼峰命名，例如：`UserService`, `OrderController`
- **方法名**：小驼峰命名，例如：`getUserById`, `createOrder`
- **数据库表名**：小写下划线，例如：`dbs_user`, `sys_role`
- **数据库字段**：小写下划线，例如：`user_id`, `create_time`

## 二、数据库约束

### 2.1 表结构约束
- **禁止删除字段**：只能新增字段，不能删除已有字段
- **禁止修改字段类型**：字段类型一旦确定，不得修改
- **必须有注释**：所有表和字段必须有中文注释
- **必须有索引**：常用查询字段必须建立索引

### 2.2 数据迁移约束
- **迁移脚本必须版本化**：所有数据库变更必须通过迁移脚本
- **迁移脚本必须可回滚**：每个迁移脚本必须包含回滚语句
- **迁移脚本必须测试**：在测试环境验证通过后才能合并

### 2.3 事务约束
- **积分操作必须事务**：所有积分相关操作必须在事务中执行
- **订单状态变更必须事务**：订单状态变更必须原子性
- **禁止长事务**：事务执行时间不得超过30秒

## 三、API接口约束

### 3.1 接口设计约束
- **RESTful风格**：所有接口必须遵循RESTful设计规范
- **统一响应格式**：所有接口必须返回统一的响应格式
```json
{
  "code": 200,
  "msg": "success",
  "data": {}
}
```
- **统一错误码**：使用预定义的错误码，不得自定义错误码

### 3.2 接口文档约束
- **必须同步更新**：修改接口必须同步更新API文档
- **必须有示例**：每个接口必须有请求和响应示例
- **必须有说明**：每个接口必须有详细的业务说明

### 3.3 接口版本约束
- **版本号规范**：接口版本号使用`v1`, `v2`格式
- **向后兼容**：新版本接口必须向后兼容
- **废弃接口标记**：废弃的接口必须标记为`@Deprecated`

### 3.4 前后端API文件分工约束
- **后端API实现归属**：后端接口入口必须放在对应业务模块的`controller`包中，DTO/VO放在对应模块的`dto`/`vo`包中；`dabashou-api`仅作为聚合启动、安全配置、全局异常、网关类配置模块，不承载业务Controller。
- **后端API契约目录**：后端接口契约说明统一维护在`docs/api/`，按模块拆分为`auth-user.md`、`skill.md`、`shelf.md`、`demand.md`、`order.md`、`point.md`、`message.md`、`credit.md`、`stat.md`、`admin.md`等文件。
- **前端API调用目录**：前端接口调用文件统一维护在`frontend/src/api/`，必须按模块拆分为`auth.ts`、`user.ts`、`skill.ts`、`shelf.ts`、`demand.ts`、`order.ts`、`point.ts`、`message.ts`、`credit.ts`、`stat.ts`、`admin.ts`等文件。
- **前端类型目录**：前端接口类型统一维护在`frontend/src/types/api.ts`或`frontend/src/types/`下的模块化类型文件，禁止在页面组件中重复定义后端响应类型。
- **接口变更同步**：任何接口路径、方法、参数、响应字段、错误码或鉴权要求变更，必须同步修改后端Controller/DTO/VO、`docs/api/`文档、`frontend/src/api/`调用文件和前端类型定义。
- **禁止散落接口**：禁止在Vue页面、组件或组合式函数中直接硬编码后端URL；禁止在后端非Controller层声明HTTP路由；禁止新增未归档到模块API文件的临时接口。
- **命名一致性**：前端API方法名、请求参数名、响应类型名必须与后端DTO/VO语义一致，例如分页统一使用`pageNum`、`pageSize`，不得混用`page`、`size`。
- **评审要求**：涉及接口的PR必须说明影响的模块、接口文件、鉴权规则、请求/响应示例和兼容性处理。

## 四、业务逻辑约束

### 4.1 订单状态机约束
- **状态不可跳变**：订单状态必须按预定义的状态机流转
- **状态变更必须记录**：每次状态变更必须记录到日志
- **超时必须处理**：所有超时状态必须有熔断机制

### 4.2 积分操作约束
- **余额不能为负**：积分余额不得为负数
- **冻结不能超额**：冻结积分不得超过可用余额
- **流水必须记录**：所有积分变动必须记录流水

### 4.3 信用评价约束
- **评价必须真实**：评价必须基于真实交易
- **评价不可修改**：评价一旦提交不可修改
- **违规必须记录**：所有违规行为必须记录存档

## 五、代码质量约束

### 5.1 代码规范
- **必须有注释**：重要业务逻辑必须有详细注释
- **必须有异常处理**：所有外部调用必须有异常处理
- **必须有日志记录**：关键操作必须有日志记录

### 5.2 测试约束
- **核心逻辑必须测试**：订单状态机、积分操作等核心逻辑必须有单元测试
- **测试覆盖率**：核心模块测试覆盖率不得低于80%
- **测试必须通过**：所有测试必须通过才能合并

### 5.3 性能约束
- **数据库查询优化**：禁止N+1查询，必须使用批量查询
- **缓存使用规范**：热点数据必须使用缓存
- **接口响应时间**：核心接口响应时间不得超过500ms

## 六、Git协作约束

### 6.1 分支管理
- **主分支保护**：`main`分支为保护分支，禁止直接推送
- **开发分支**：所有开发在`develop`分支进行
- **功能分支**：每个功能使用独立的功能分支

### 6.2 提交规范
- **提交信息格式**：`[模块] 类型: 描述`
  - 示例：`[user] feat: 添加用户注册功能`
  - 示例：`[order] fix: 修复订单状态流转bug`
- **提交粒度**：每个提交只包含一个功能或修复
- **提交前必须测试**：提交前必须运行测试并确保通过

### 6.3 代码审查
- **必须审查**：所有代码合并必须经过审查
- **审查重点**：业务逻辑正确性、安全性、性能
- **审查记录**：审查意见必须记录在PR中

## 七、安全约束

### 7.1 数据安全
- **密码加密**：用户密码必须加密存储
- **敏感信息脱敏**：日志中敏感信息必须脱敏
- **SQL注入防护**：所有SQL必须使用参数化查询

### 7.2 接口安全
- **身份认证**：所有接口必须进行身份认证
- **权限控制**：接口必须进行权限控制
- **防重放攻击**：关键接口必须防重放

### 7.3 业务安全
- **防刷机制**：积分获取、评价等必须有防刷机制
- **风控机制**：异常行为必须有风控机制
- **审计日志**：关键操作必须记录审计日志

## 八、文档约束

### 8.1 必须维护的文档
- **README.md**：项目说明文档
- **AGENTS.md**：协作约束文档
- **API文档**：接口文档
- **数据库设计文档**：数据库表结构文档

### 8.2 文档更新规范
- **同步更新**：代码变更必须同步更新文档
- **版本管理**：文档必须有版本管理
- **变更记录**：文档变更必须记录

## 九、禁止事项

### 9.1 绝对禁止
- ❌ 禁止直接修改数据库表结构
- ❌ 禁止删除他人的代码
- ❌ 禁止绕过状态机直接修改订单状态
- ❌ 禁止在生产环境调试代码
- ❌ 禁止提交敏感信息（密码、密钥等）

### 9.2 强烈不推荐
- ⚠️ 不推荐使用`SELECT *`查询
- ⚠️ 不推荐在循环中执行数据库操作
- ⚠️ 不推荐使用魔法数字
- ⚠️ 不推荐忽略异常处理

## 十、违规处理

### 10.1 轻微违规
- 代码格式不规范
- 注释不完整
- 文档未更新

**处理方式**：要求修改后重新提交

### 10.2 严重违规
- 破坏模块边界
- 破坏数据库约束
- 破坏业务逻辑

**处理方式**：拒绝合并，要求重新设计

### 10.3 极其严重违规
- 删除他人代码
- 绕过安全机制
- 提交恶意代码

**处理方式**：禁止参与项目开发
