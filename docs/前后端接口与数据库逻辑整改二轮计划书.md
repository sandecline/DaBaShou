# 搭把手前后端接口与数据库逻辑整改二轮计划书

**日期**：2026年7月1日  
**目标**：在第一轮已完成接口路径、公开用户详情、异常契约、前端 API 分工和基础构建验证的基础上，继续补齐后台管理接口、完善数据库索引与迁移脚本、强化权限闭环、补足集成测试和联调验收。

## 一、二轮整改原则

1. 不推倒重来，继续保留现有 `backend/dabashou-*` 多模块结构。
2. 后台业务 Controller 放在 `dabashou-admin` 模块，`dabashou-api` 仍只负责启动、安全、全局异常、过滤器和聚合配置。
3. 涉及数据库结构变更时，只通过新增 Flyway 迁移脚本处理，不直接改表，不删除字段，不修改既有字段类型。
4. 后台接口先以最小可用闭环为目标，优先保证列表、详情、状态处理、配置读写可用。
5. 所有接口变更必须同步维护后端 Controller/DTO/VO、`docs/api/`、`frontend/src/api/` 和前端类型定义。
6. 所有业务异常必须交给 `GlobalExceptionHandler`，禁止 Controller 局部捕获 `BusinessException` 后返回 HTTP 200。
7. 管理端权限必须前后端双重校验：前端路由拦截只是体验兜底，后端仍必须使用角色鉴权。

## 二、一轮已完成基线

### 2.1 已完成内容

- 新增 `GET /api/v1/users/{userId}` 公开脱敏用户详情接口。
- 用户小铺接口统一为 `GET /api/v1/shelves/users/{userId}/shelves`。
- 需求接单统一为 `acceptDemand` 和 `POST /api/v1/demands/{id}/accept`。
- 前端订单接口统一调用 `/v1/orders/**`，后端继续兼容 `/api/v1/order/**`。
- `auth.ts` 回归认证职责，新增 `frontend/src/api/user.ts` 承载用户资料、校园认证、信任分、公开用户详情。
- 移除已扫描到的 Controller 局部 `catch (BusinessException)`。
- 更新 `docs/api/auth-user.md`、`docs/api/global.md`、`docs/api/order.md`、`docs/api/shelf.md`。
- 前端增加 `ADMIN` 角色路由守卫。
- `mvn test` 通过，`npm run build` 通过。

### 2.2 二轮必须接续处理的问题

- `dabashou-admin` 仍缺少后台管理 Controller。
- 后台接口虽有前端调用和文档，但后端未形成完整闭环。
- 管理员角色来源、账号分配和后端鉴权需要确认。
- 数据库索引仍需系统核查，部分唯一索引或复合索引需要数据清洗后再补。
- 管理端页面、API、文档和后端返回字段仍需逐项联调。
- 订单/积分/评价等核心链路还需要更细的异常状态测试。

## 三、二轮目标接口

### 3.1 后台用户管理

- `GET /api/admin/v1/users`
- `GET /api/admin/v1/users/{id}`
- `PUT /api/admin/v1/users/{id}/status`
- `POST /api/admin/v1/users/{id}/reset-password`

要求：

- 仅管理员可访问。
- 列表支持 `keyword`、`status`、`pageNum`、`pageSize`。
- 手机号必须脱敏返回。
- 重置密码必须生成随机临时密码并加密入库。
- 禁用用户后，后续登录必须被拒绝。

### 3.2 后台订单管理

- `GET /api/admin/v1/orders`
- `GET /api/admin/v1/orders/{id}`
- `POST /api/admin/v1/orders/{id}/arbitrate`

要求：

- 列表支持 `keyword`、`status`、`pageNum`、`pageSize`。
- 详情返回买家、卖家、货架、需求、积分金额、状态、争议信息。
- 仲裁接口复用订单模块状态机，不允许绕过状态流转。
- 仲裁结果必须产生审计日志。

### 3.3 后台信用管理

- `GET /api/admin/v1/violations`
- `POST /api/admin/v1/violations/{id}`
- `GET /api/admin/v1/appeals`
- `POST /api/admin/v1/appeals/{id}`
- `GET /api/admin/v1/reviews`
- `DELETE /api/admin/v1/reviews/{id}`

要求：

- 违规处理必须能追溯目标用户、举报人和订单。
- 申诉处理必须绑定违规记录。
- 删除评价建议先做软删除或隐藏，不直接物理删除。
- 信用分变更必须记录流水。

### 3.4 后台校园认证

- `GET /api/admin/v1/campus-auths`
- `POST /api/admin/v1/campus-auths/{id}`

要求：

- 支持按认证状态筛选。
- 审核通过后同步更新用户认证状态或认证标识。
- 审核拒绝必须记录原因。

### 3.5 系统配置

- `GET /api/admin/v1/config`
- `PUT /api/admin/v1/config`

要求：

- 后端使用 `sys_config` 表。
- 配置更新必须按白名单校验 key，禁止任意写入危险配置。
- 敏感配置不得明文返回。

### 4.2 建议补齐索引

二轮先核查现有索引和数据重复情况，再决定是否新增迁移。

- `dbs_user.phone` 唯一索引：新增前必须检查空值、重复手机号和测试数据。
- `dbs_skill_shelf(user_id, status)` 复合索引。
- `dbs_skill_shelf(skill_tag_id, status)` 复合索引。
- `dbs_demand(user_id, status)` 复合索引。
- `dbs_demand(skill_tag_id, status)` 复合索引。
- `dbs_order(buyer_id, status)` 复合索引。
- `dbs_order(seller_id, status)` 复合索引。
- `dbs_point_transaction(user_id, create_time)` 已存在，二轮确认 H2 与 MySQL 迁移一致。
- `dbs_chat_message(session_id, create_time)` 复合索引。
- `sys_notification(user_id, is_read, create_time)` 复合索引。

### 4.3 迁移脚本要求

- 生产脚本放在 `backend/dabashou-api/src/main/resources/db/migration/`。
- 测试 H2 脚本放在 `backend/dabashou-api/src/test/resources/db/migration-h2/`。
- 新迁移建议版本号从当前最大版本后递增，例如 `V1.16.0__admin_indexes_and_constraints.sql`。
- MySQL 与 H2 脚本必须同时维护。
- 每个新增索引必须先验证现有数据是否满足约束。
- 唯一索引迁移必须包含数据清洗说明和回滚方案文档。

## 五、权限与安全整改计划

### 5.1 后端鉴权

- `/api/admin/v1/**` 必须统一要求管理员角色。
- 后台 Controller 方法建议加 `@PreAuthorize("hasRole('ADMIN')")`。
- Spring Security 配置必须确认角色前缀和 JWT `roles` claim 一致。
- 普通用户访问后台接口必须返回 HTTP 403 + `code=403`。

### 5.2 登录与账号状态

- 禁用用户不得登录。
- 用户状态异常时刷新 token 应失败。
- 后续可增加 token 黑名单或 token version 机制，使禁用立即生效。

### 5.3 审计日志

二轮至少覆盖：

- 用户启用/禁用。
- 重置密码。
- 订单仲裁。
- 违规处理。
- 申诉处理。
- 系统配置更新。

审计日志建议写入 `sys_log`，记录操作人、操作类型、目标资源、请求摘要、结果和耗时。

## 十、风险清单

| 风险 | 影响 | 二轮处理策略 |
|------|------|--------------|
| 生产手机号存在重复 | 无法直接加唯一索引 | 先输出重复数据检查 SQL，清洗后再加唯一索引 |
| 后台接口跨模块直接查表过多 | 模块边界变弱 | 二轮可用 JdbcTemplate 最小闭环，三轮抽 Facade/API |
| 管理员角色数据不完整 | 后台无法访问 | 检查 `sys_role`、`sys_user_role` 初始化数据 |
| 信用表字段与前端展示不完全一致 | 页面字段为空或报错 | 补 VO 映射，必要时新增字段迁移 |
| 删除评价物理删除风险高 | 数据审计缺失 | 二轮优先软删除或隐藏状态，不做物理删除 |
| 配置接口任意 key 写入 | 安全风险 | 使用白名单和敏感字段脱敏 |

## 十一、三轮预留方向

- 模块 API/Facade 抽象，减少跨模块直接依赖。
- 订单超时任务和熔断机制。
- 积分流水强一致性审计。
- 后台操作审计可视化。
- API 自动化契约测试。
- 前端大 chunk 拆包优化。
