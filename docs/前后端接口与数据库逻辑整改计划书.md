# 搭把手前后端接口与数据库逻辑整改计划书

**日期**：2026年7月1日  
**目标**：统一前后端接口路径、统一命名逻辑、打通核心业务链路、修复当前主要结构问题，并系统检查数据库表结构、状态字段、事务与数据一致性。

## 一、整改原则

1. 不推倒重来，保留现有 `backend/dabashou-*` 多模块结构。
2. 后端接口入口仍放在各业务模块 `controller` 包中，`dabashou-api` 只负责启动、安全、全局异常、过滤器和聚合配置。
3. 前端接口调用统一维护在 `frontend/src/api/`，页面和组件禁止直接硬编码后端 URL。
4. 统一采用 `/api/v1/{module}/**` 风格；前端请求层 baseURL 为 `/api` 时，前端 API 文件内写 `/v1/**`。
5. 分页参数统一使用 `pageNum`、`pageSize`；排序统一使用 `sortBy`。
6. 错误响应统一为 HTTP 状态码和 `AjaxResult.code` 同步表达，例如 HTTP 401 + `code=401`。
7. 数据库结构只通过迁移脚本演进，不直接改表，不删除字段，不修改既有字段类型。

## 二、当前主要问题

### 2.1 前后端接口路径不一致

需要优先修复以下已发现问题：

- 用户小铺接口：
  - 前端：`/v1/users/{userId}/shelves`
  - 后端：`/api/v1/shelves/users/{userId}/shelves`
  - 需要统一为一个路径，建议后端保留 `/api/v1/shelves/users/{userId}/shelves`，前端同步调整。

- 需求接单接口：
  - 前端：`/v1/demands/{id}/bid`
  - 后端：`/api/v1/demands/{id}/accept`
  - 建议统一命名为 `acceptDemand`，路径为 `/api/v1/demands/{id}/accept`。

- 公开用户详情接口：
  - 前端：`/v1/users/{userId}`
  - 后端目前缺少对应公开 Controller。
  - 建议新增后端公开只读接口：`GET /api/v1/users/{userId}`，返回脱敏用户资料。

- 后台管理接口：
  - 前端已有 `/admin/v1/users`、`/admin/v1/orders`、`/admin/v1/violations`、`/admin/v1/config` 等调用。
  - 后端目前主要只有 `/api/admin/v1/stats/**`。
  - 需要补齐后台 Controller，或先隐藏对应前端入口。

### 2.2 命名逻辑不统一

- `bidDemand` 与后端 `accept` 语义不一致。
- `SkillShelf`、`ShelfItemVo`、`Demand`、`DemandItemVo` 等旧别名和后端 VO 命名并存。
- `page/size/sort` 与 `pageNum/pageSize/sortBy` 混用。
- `/api/v1/order` 与 `/api/v1/orders` 同时存在，建议最终统一为复数 `/api/v1/orders`，旧路径作为兼容入口逐步废弃。

### 2.3 统一异常契约仍不彻底

多个 Controller 内部仍捕获 `BusinessException` 后返回 `AjaxResult.fail(...)`，这会导致业务错误变成 HTTP 200。需要移除 Controller 局部 `try/catch BusinessException`，统一交给 `GlobalExceptionHandler`。

### 2.4 模块边界仍偏重直接依赖

`dabashou-order` 直接依赖 `point/user/skill/shelf/demand`，`credit` 依赖 `order/user`，`stat` 依赖 `order/point/user`。短期可以保留，但需要逐步收敛为模块 API 或 Facade，避免跨模块直接调用业务实现。

## 四、前端整改计划

### 4.1 API 文件整理

- `frontend/src/api/auth.ts`：只保留登录、注册、刷新、退出、个人资料接口。公开用户详情建议拆到 `user.ts`。
- 新增或完善 `frontend/src/api/user.ts`：公开用户详情、个人资料、校园认证、信任分。
- `frontend/src/api/shelf.ts`：修正用户货架路径为 `/v1/shelves/users/{userId}/shelves`。
- `frontend/src/api/demand.ts`：将 `bidDemand` 改为 `acceptDemand`，路径使用 `/v1/demands/{id}/accept`。
- `frontend/src/api/order.ts`：统一使用 `/v1/orders/**`，保留 `/v1/order/**` 兼容期不超过一轮版本。
- `frontend/src/api/admin.ts`：按后端实际 Controller 补齐，未实现的接口不要在页面中直接调用。
- 保留 `_params.ts`，所有分页接口统一通过 `normalizePageParams` 转换。

### 4.2 类型命名整理

- 以后端 VO/DTO 为准，保留兼容别名但标注迁移方向。
- 页面组件逐步从 `SkillShelf` 迁移到 `ShelfItemVo`。
- 页面组件逐步从 `Demand` 迁移到 `DemandItemVo` 或 `DemandDetailVo`。
- 页面组件逐步从 `OverviewStat` 迁移到 `PersonalOverviewVo`。

### 4.3 页面联调重点

- 首页未登录状态不得请求受保护接口。
- 登录成功后不得依赖非关键资料接口阻塞跳转。
- 技能列表、需求看板、详情页必须无需登录可访问。
- 我的订单、消息、积分、信用中心必须登录后访问。
- 后台页面必须校验管理员权限，不具备后端接口前先隐藏入口或展示未开放状态。

## 五、后端整改计划

### 5.1 路径统一

- 统一订单路径为 `/api/v1/orders/**`。
- `OrderController` 暂时保留 `/api/v1/order/**` 兼容，但标注废弃。
- 新增公开用户详情接口 `GET /api/v1/users/{userId}`。
- 统一需求接单为 `POST /api/v1/demands/{id}/accept`。
- 与前端同步更新 `docs/api/` 文档。

### 5.2 异常统一

- 移除 Controller 内 `catch (BusinessException e)` 后 `AjaxResult.fail(...)` 的写法。
- 所有业务异常直接抛出，由 `GlobalExceptionHandler` 返回对应 HTTP 状态码。
- `404`、`409`、`403`、`401` 必须用 HTTP 状态码同步表达。

### 5.3 后台接口补齐

优先补齐：

1. 用户管理：列表、详情、启用/禁用、重置密码。
2. 订单管理：列表、详情、争议仲裁。
3. 信用管理：违规列表、处理违规、申诉列表、处理申诉。
4. 系统配置：读取配置、更新配置。

### 5.4 模块边界优化

- 短期：保留 Maven 直接依赖，但明确只允许依赖对外 API/Facade，不直接操作对方内部 Service/Mapper。
- 中期：为 `point`、`user`、`shelf`、`demand` 提供模块 API 接口。
- 长期：`order` 只依赖模块 API，不直接依赖业务模块实现细节。

## 六、数据库逻辑检查计划

### 6.1 表结构检查

重点检查：

- 用户表是否包含账号状态、软删除字段、唯一索引。
- 货架表是否有 `user_id`、`skill_tag_id`、`status`、价格、时长、上下架时间。
- 需求表是否有 `user_id`、`skill_tag_id`、`status`、悬赏积分、截止时间。
- 订单表是否完整覆盖买家、卖家、货架、需求、金额、状态、核销码、取消/完成时间。
- 积分账户表是否有可用余额、冻结余额、总收入、总支出。
- 积分流水表是否能追踪订单、操作类型、变更前后余额。
- 评价表是否绑定订单、评价人、被评价人，并限制重复评价。
- 违规与申诉表是否能串联订单、举报人、被举报人、处理结果。

### 6.2 索引检查

建议补齐或确认：

- `dbs_user.username` 唯一索引。
- `dbs_user.phone` 唯一索引。
- `dbs_skill_shelf.user_id`、`skill_tag_id`、`status` 索引。
- `dbs_demand.user_id`、`skill_tag_id`、`status` 索引。
- `dbs_order.buyer_id`、`seller_id`、`status`、`order_no` 索引。
- `point_transaction.user_id`、`order_id`、`type`、`create_time` 索引。
- `message/chat` 相关表按会话、发送人、接收人、创建时间建索引。

### 6.3 状态机检查

订单状态必须按以下顺序流转：

- `1 待支付`
- `2 已支付/担保中`
- `3 服务中`
- `4 待确认`
- `5 已完成`
- `0 已取消`
- `6 已退款`
- `7 争议中`

必须禁止：

- 未支付直接开始服务。
- 已取消订单继续支付。
- 非买家确认订单。
- 非卖家开始服务。
- 非买卖双方访问订单详情。
- 已完成订单重复结算积分。

### 6.4 积分一致性检查

必须保证：

- 支付时冻结积分在同一事务内完成。
- 取消订单时解冻积分。
- 完成订单时冻结积分转入卖家余额。
- 退款时冻结积分返还买家。
- 余额不能为负。
- 冻结积分不能超过可用余额。
- 每一次积分变化必须有流水。

### 6.5 评价与信用检查

必须保证：

- 评价只能基于真实订单。
- 只有订单完成后才能评价。
- 同一订单同一评价人只能评价一次。
- 评价提交后不可修改。
- 违规举报必须能追溯订单或目标用户。
- 申诉必须绑定违规记录。

### 6.6 迁移脚本检查

- 生产迁移脚本必须放在 `backend/dabashou-api/src/main/resources/db/migration/`。
- 测试 H2 迁移脚本必须放在 `backend/dabashou-api/src/test/resources/db/migration-h2/`。
- 新增字段只通过新迁移脚本处理。
- 不删除字段，不直接改字段类型。
- 每次新增迁移后必须跑后端测试。

## 七、测试计划

### 7.1 后端测试

- 未登录访问个人资料返回 HTTP 401。
- 普通用户访问后台接口返回 HTTP 403。
- 公开技能分类、技能标签、货架列表、需求看板无需 token。
- 登录后获取个人资料成功。
- 用户小铺公开详情成功。
- 需求接单接口成功。
- 需求接单非法状态返回 HTTP 409。
- 订单支付、开始服务、核销、确认完整链路通过。
- 重复支付、重复确认、非法退款返回 HTTP 409。
- 积分余额不足支付失败，返回 HTTP 409 或 400。

### 7.2 前端验证

- `npm run build` 必须通过。
- 未登录首页不再弹未授权。
- 登录按钮点击后能进入首页或 redirect 页面。
- 用户小铺公开页面可加载。
- 需求详情页接单按钮调用正确接口。
- 后台菜单未授权时不可访问。
- token 过期后跳转登录并保留 redirect。

## 八、执行顺序

1. 修正前后端路径不一致问题。
2. 补公开用户详情接口。
3. 修正需求接单命名和路径。
4. 移除 Controller 局部业务异常捕获。
5. 补后台管理 Controller 或隐藏未实现入口。
6. 检查数据库索引、状态字段和迁移脚本。
7. 补集成测试。
8. 跑 `mvn test` 和 `npm run build`。
9. 同步更新 `docs/api/` 与前端 API README。
