# 搭把手 工程约束文档

## 一、总则

### 1.1 适用范围
本文件是搭把手项目的**工程约束规范**，适用于所有参与开发的人员和 Agent。所有代码提交必须同时满足本文件与 `AGENTS.md` 的要求。

### 1.2 与 AGENTS.md 的关系
- `AGENTS.md`：面向 AI Agent 的**协作规约**（提交格式、PR 模板、模块边界等协作层面的约定）
- `docs/constraints.md`（本文件）：面向所有开发者和 Agent 的**工程约束**（技术栈、数据库、API、测试、安全等工程层面的规范）
- 两者互为补充，冲突时以本文件为准

### 1.3 优先级
本项目遵循以下优先级：**安全约束 > 业务逻辑约束 > 数据库约束 > API 约束 > 代码质量约束 > 样式约束**

## 二、技术栈版本锁定

| 层面   | 技术选型          | 版本       | 备注                   |
| ------ | ----------------- | ---------- | ---------------------- |
| JDK    | OpenJDK / GraalVM | **21**     | 最低版本，不得降级     |
| 框架   | Spring Boot       | **4.0.x**  | 含 Web、WebSocket、AOP |
| ORM    | MyBatis-Plus      | **3.5.x**  | 禁止使用 MyBatis XML  |
| 缓存   | Redis             | **7.x**    | Spring Data Redis      |
| 数据库 | MySQL             | **8.0+**   | InnoDB 引擎            |
| 前端   | Vue               | **3.4+**   | Composition API        |
| UI库   | Element Plus      | **2.x**    | 仅管理后台使用         |
| 构建   | Vite              | **5.x**    | 前端构建工具           |
| 语言   | TypeScript        | **5.x**    | 前端全量 TS            |
| 状态   | Pinia             | **2.x**    | 前端状态管理           |
| 样式   | Tailwind CSS      | **3.x**    | 原子化 CSS             |

### 2.1 版本约束
- **禁止擅自升级/降级**主版本号，升级必须提交 RFC 并通过评审
- `pom.xml` 中所有依赖必须显式声明版本号（不依赖 Spring Boot BOM 的传递版本除外）
- 前端 `package.json` 中依赖版本使用 `~` 锁定补丁版本，禁止使用 `*`
- 每次版本变更必须在 `migration/` 目录附变更说明

### 2.2 禁止引入的技术
- **Lombok**：使用手写 getter/setter/builder，提高可读性
- **Swagger/SpringFox**：使用 Knife4j 或 SpringDoc 替代
- **任何 alpha/beta 依赖**：生产代码不得依赖预发布版本

## 三、目录与命名

### 3.1 后端命名
- **包名**：`com.dabashou.{module}`，例如：`com.dabashou.user`
- **Controller**：`XxxController`
- **Service 接口**：`XxxService`
- **Service 实现**：`XxxServiceImpl`
- **Mapper 接口**：`XxxMapper`
- **实体类**：`Xxx`（如 `User`、`Order`）
- **DTO**：`XxxDto`（请求入参）
- **VO**：`XxxVo`（响应出参）

### 3.2 数据库命名
- **表名**：小写下划线，`dbs_` 前缀为业务表，`sys_` 前缀为系统表
- **字段名**：小写下划线，如 `user_id`、`create_time`
- **索引名**：`idx_表名_字段`，如 `idx_user_campus`
- **外键名**：`fk_表名_关联表`，如 `fk_order_user`

### 3.3 前端命名
- **组件文件**：大驼峰，如 `SkillCard.vue`
- **组合式函数**：`useXxx.ts`，如 `useUser.ts`
- **Store 文件**：`xxxStore.ts`，如 `userStore.ts`
- **API 文件**：`xxxApi.ts`，如 `userApi.ts`
- **路由文件**：`xxxRoutes.ts`，如 `skillRoutes.ts`
- **TypeScript 类型**：大驼峰接口名，如 `interface UserInfo {}`

### 3.4 API 路径命名
- 统一小写中划线：`/api/user-profile`，不用驼峰
- RESTful 风格：`GET /api/shelf/{id}`，`POST /api/shelf`，`PUT /api/shelf/{id}`，`DELETE /api/shelf/{id}`

## 四、模块边界

### 4.1 依赖方向（只允许单向）
```
dabashou-api（聚合入口，只依赖其他模块）
  ├── dabashou-admin（管理后台，依赖 system/credit/stat）
  ├── dabashou-order（订单，依赖 point/user/skill/shelf/demand）
  ├── dabashou-point（积分，依赖 common）
  ├── dabashou-credit（信用，依赖 order/user）
  ├── dabashou-shelf（货架，依赖 skill/user）
  ├── dabashou-demand（需求，依赖 skill/user）
  ├── dabashou-skill（技能，依赖 common）
  ├── dabashou-message（消息，依赖 user/order）
  ├── dabashou-stat（统计，依赖 order/point/user）
  ├── dabashou-user（用户，依赖 system/common）
  ├── dabashou-system（系统，依赖 common）
  └── dabashou-common（基础，不依赖任何业务模块）
```

### 4.2 硬性约束
- `dabashou-common` **禁止包含业务逻辑**，只放工具类、常量、异常定义、公共组件
- `dabashou-api` **只负责聚合**，不写业务代码，不定义实体
- **禁止跨模块直接调用 Service**，模块间必须通过 API 接口或事件机制通信
- 新增模块必须走 RFC 审批流程

## 五、数据库约束

### 5.1 表结构约束
- **禁止删除字段**：只允许新增字段，如需废弃则添加 `deprecated` 注释
- **禁止修改字段类型**：字段类型一旦上线不得修改，如需变更则新增字段 + 数据迁移
- **必须有注释**：所有表和字段必须有中文 COMMENT
- **必须有索引**：所有外键字段、高频查询字段必须建立索引
- **必须有主键**：统一使用 `BIGINT AUTO_INCREMENT` 自增主键
- **时间字段**：统一使用 `DATETIME` 类型，包含 `create_time` 和 `update_time`

### 5.2 迁移机制
- **工具**：Flyway（Spring Boot 原生集成）
- **脚本命名**：`V{主版本}.{子版本}.{修订}__{描述}.sql`，例如：`V1.0.0__init_user_skill_tables.sql`
- **脚本位置**：`database/migration/`
- **禁止修改已执行的脚本**：已合并到 develop 的迁移脚本视为不可变
- **必须包含回滚说明**：每个迁移脚本在 PR 描述中附回滚 SQL
- **必须在测试环境验证**：迁移脚本先在本地 + 测试环境执行通过，再合并

### 5.3 事务约束
- **积分操作必须事务**：所有涉及 `dbs_point_transaction` + `dbs_user.point_balance` 的操作必须在 `@Transactional` 中执行
- **订单状态变更必须事务**：`dbs_order.status` 变更 + 关联积分操作必须原子执行
- **禁止长事务**：事务执行时间不得超过 30 秒，超时自动回滚
- **禁止在事务中做远程调用**：HTTP/RPC 调用不得在事务内执行

## 六、API 约束

### 6.1 统一响应格式
```json
{
  "code": 200,
  "msg": "success",
  "data": {}
}
```
- 成功：`code=200`
- 业务异常：`code=4xx`（参见附录错误码）
- 系统异常：`code=500`

### 6.2 统一错误码（详见附录）
- 使用项目预定义的错误码，禁止自行定义
- 前端根据 `code` 做通用拦截处理（401 跳登录、403 提示无权限、500 提示系统错误）

### 6.3 分页规范
- 请求参数：`pageNum`（页码，从 1 开始）、`pageSize`（每页条数，默认 10，最大 100）
- 响应格式：
```json
{
  "total": 100,
  "list": [],
  "pageNum": 1,
  "pageSize": 10
}
```

### 6.4 接口版本
- URL 前缀：`/api/v1/xxx`、`/api/v2/xxx`
- 向后兼容：新版本接口必须兼容旧版本请求参数
- 废弃接口：使用 `@Deprecated` 注解 + 3 个版本的过渡期

### 6.5 接口文档
- 使用 **Knife4j**（SpringDoc/OpenAPI 3）自动生成
- Controller 方法必须有 `@Operation` + `@Parameter` 注解
- 接口变更必须同步更新文档注解

## 七、业务逻辑约束

### 7.1 订单状态机（8 状态，不可跳变）
| 状态码 | 状态名 | 允许的下一状态                    |
| ------ | ------ | --------------------------------- |
| 0      | 已取消 | 无（终态）                        |
| 1      | 待支付 | → 已支付(2)、已取消(0)            |
| 2      | 已支付 | → 服务中(3)、已退款(6)、已取消(0) |
| 3      | 服务中 | → 待确认(4)、已退款(6)、已取消(0) |
| 4      | 待确认 | → 已完成(5)、争议中(7)、已退款(6) |
| 5      | 已完成 | 无（终态）                        |
| 6      | 已退款 | → 已取消(0)                       |
| 7      | 争议中 | → 已完成(5)、已退款(6)            |

- 每次状态变更必须写入日志/审计记录
- 超时熔断：待支付 15 分钟 → 自动取消；核销码 30 分钟 → 自动退款；待确认 7 天 → 自动完成

### 7.2 积分操作约束
- `point_balance`（可用余额）不得为负数
- 冻结积分不得超过当前可用余额
- 每次变动必须同步写入 `dbs_point_transaction`
- 退改扣分：买家取消扣 10%，卖家取消扣 20%

### 7.3 信用评价约束
- 评价必须基于已完成的订单（`status=5`）
- 评价一旦提交不可修改、不可删除
- 违规行为必须记录到 `credit_violation`
- 信任分自动调整：好评 +0.1，差评 -0.3，放鸽子 -1.0

## 八、代码质量

### 8.1 注释规范
- 类必须有 Javadoc 说明用途
- 公共方法必须有 Javadoc（参数、返回值、异常）
- 复杂业务逻辑必须有行内注释说明意图
- 前端组件必须有 Props 和 Emits 的 TypeScript 类型注释

### 8.2 异常处理
- 所有外部调用（Redis、HTTP、DB）必须有 try-catch
- 禁止吞掉异常（catch 块中必须 log 或 rethrow）
- Controller 层禁止直接 catch，统一由全局异常处理器处理
- 前端 API 请求统一在 `request.ts` 中做错误拦截

### 8.3 日志规范
- 日志级别：`ERROR`（系统异常）、`WARN`（业务异常）、`INFO`（关键操作）、`DEBUG`（调试信息）
- 关键操作必须 INFO 级别日志：登录、下单、支付、核销、退款
- 日志中禁止输出密码、token、手机号等敏感信息（必须脱敏）
- 审计日志独立存储（`sys_log` 表），不与应用日志混合

### 8.4 性能约束
- 禁止 N+1 查询，必须使用批量查询或 JOIN
- 热点数据（用户信息、技能分类）必须使用 Redis 缓存
- 核心接口响应时间 ≤ 500ms
- 分页查询禁止 `SELECT *`，必须指定字段

## 十、Git 协作

### 10.1 分支管理
```
main          ← 生产保护分支，禁止直接推送
  └── develop ← 开发基线分支
       ├── feature/xxx    ← 功能分支
       ├── fix/xxx        ← Bug 修复分支
       └── hotfix/xxx     ← 紧急修复分支
```

### 10.2 提交规范
- 格式：`[模块] 类型: 描述`
- 类型：`feat`（新功能）、`fix`（修复）、`refactor`（重构）、`docs`（文档）、`test`（测试）、`chore`（杂项）
- 示例：`[user] feat: 添加校园认证接口`、`[order] fix: 修复状态机跳变问题`

### 10.3 PR 要求
- 标题遵循提交规范
- 描述中说明变更内容、影响范围、测试方式
- 至少 1 人审核通过
- CI 必须通过（lint + typecheck + test）

## 十一、安全约束

### 11.1 数据安全
- 用户密码使用 **BCrypt** 加密存储，禁止明文或 MD5
- 手机号、邮箱在日志和响应中必须脱敏（如 `138****1234`）
- 所有 SQL 必须使用参数化查询（MyBatis-Plus 自动防注入），禁止字符串拼接

### 11.2 接口安全
- 所有 `/api/**` 接口必须经过 JWT 认证（除注册/登录等公开接口）
- 管理接口（`/api/admin/**`）必须校验管理员角色
- 关键接口（下单、支付、核销）必须防重放（幂等 Token）

### 11.3 业务安全
- 积分获取必须有防刷机制（注册奖励仅限一次、每日签到上限）
- 评价必须基于真实交易，禁止自评
- 异常行为（短时间大量下单/取消）触发风控告警
- 审计日志独立存储，保留 180 天

## 十二、密钥与环境变量

### 12.1 配置分离
- `.env` 文件加入 `.gitignore`，禁止提交到仓库
- 使用 Spring Profile 区分环境：`application-dev.yml`、`application-prod.yml`
- 敏感配置（数据库密码、Redis 密码、JWT Secret）通过环境变量注入

### 12.2 禁止事项
- 禁止在代码中硬编码密码、密钥、Token
- 禁止在日志中输出敏感配置
- 禁止将 `.env`、`application-prod.yml` 提交到 Git

## 十三、日志规范

### 13.1 日志级别使用
| 级别  | 场景                                    |
| ----- | --------------------------------------- |
| ERROR | 系统异常（DB 连接失败、Redis 不可用）   |
| WARN  | 业务异常（参数错误、余额不足、状态冲突）|
| INFO  | 关键操作（登录、下单、支付、核销、退款）|
| DEBUG | 调试信息（方法入参、SQL 语句）          |

### 13.2 日志格式
- 包含：时间戳、日志级别、类名、方法名、traceId、业务ID
- 脱敏：密码字段输出 `***`，手机号输出 `138****1234`

### 13.3 审计日志
- 独立写入 `sys_log` 表
- 记录：操作人、操作类型、操作内容、IP 地址、操作时间
- 关键操作：用户注册/登录、订单状态变更、积分变动、系统配置变更

## 十五、禁止事项与违规处理

### 15.1 绝对禁止（拒绝合并）
- ❌ 直接修改数据库表结构（必须通过 Flyway 迁移脚本）
- ❌ 删除他人代码（必须通过 PR 沟通）
- ❌ 绕过订单状态机直接修改状态
- ❌ 在生产环境调试代码
- ❌ 提交敏感信息（密码、密钥、token）

### 15.2 严重违规（要求重新提交）
- ⚠️ 破坏模块边界（跨模块直接调用 Service）
- ⚠️ 破坏数据库约束（删字段、改类型、无注释）
- ⚠️ 核心逻辑缺少单元测试
- ⚠️ 事务缺失（积分/订单操作无 @Transactional）

### 15.3 轻微违规（要求修改）
- 代码格式不规范
- 注释不完整
- 文档未更新
- 命名不符合规范
